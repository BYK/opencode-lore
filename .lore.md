## Long-term Knowledge

### Architecture

<!-- lore:019e1320-fe02-7a60-b34a-e3e0a83b5e54 -->
* **Distillation discovery via periodic temporal adapter polling**: Distillation service runs background polling loop calling \`findUndistilledSessions()\` on temporal adapter every 30 seconds. Discovery queries sessions where \`distilled = 0\` and message count meets minimum threshold. Both file-based (temporal-fs.ts) and database (temporal-db.ts) adapters implement this interface. Sessions become discoverable immediately when messages are added, but only marked complete after successful knowledge extraction via \`markSessionDistilled()\`.

<!-- lore:019e1323-ffdb-7556-9624-5fa69af1afec -->
* **urgentDistillation singleton causes cross-session race conditions**: urgentDistillation is module-level singleton in gradient.ts with consume-on-read pattern. needsUrgentDistillation() returns current value then resets to false. Three async consumers (gateway pipeline.ts:1067, opencode index.ts:703, distillation.ts:603) race for the flag across all sessions in same process. First caller gets true, others get false even if their session needs distillation. Requires per-session tracking to fix race condition.

### Gotcha

<!-- lore:019e1322-eb6a-7660-8975-c1d2e91096cf -->
* **fastembed ARM64 compatibility via tokenizers override**: fastembed 2.1.0 pins \`@anush008/tokenizers\` to \`^0.0.0\` (exactly 0.0.0), breaking ARM64 support which requires 0.5.0+. Override with \`"resolutions": { "@anush008/tokenizers": "^0.5.0" }\` in package.json. Clear vendor staging cache after changing overrides: \`rm -rf /tmp/fastembed\_cache\` or \`rm -rf ~/.cache/fastembed\` to force fresh download. The 0.5.0 release added prebuilt ARM64 binaries without API changes - fastembed only uses \`Tokenizer.fromFile()\` and \`AddedToken\` constructor which remain stable.

<!-- lore:019e1325-1833-71f9-8c48-ea8eb2d00c7d -->
* **Gateway creates temporal sessions for every request including sub-agents**: Gateway creates new sessions for every request including opencode sub-agent calls, causing temporal message storage bloat. Each \`stream=false\` sub-agent request (explore, plan, code agents) gets treated as new session with temporal.recordMessage() call. Unlike opencode adapter which has shouldSkip() logic for child sessions, gateway has no filtering. Sub-agents make single API calls that appear as \`new=true messages=1\` in logs but never continue, creating phantom sessions that accumulate in temporal storage and distillation queue.

<!-- lore:019e1325-1824-7b1d-ba5f-b229212ee390 -->
* **Gateway treats sub-agent sessions as new sessions causing excessive distillation**: Gateway pipeline treats every opencode sub-agent request as new session because it lacks shouldSkip() logic present in opencode adapter. Sub-agents (explore, code, plan) make single API calls with stream=false, get unique fingerprints, appear as new=true sessions that never continue. Each gets temporal messages stored and scheduled for distillation despite being ephemeral tool calls. OpenCode adapter filters these via shouldSkip(sessionId) checking child session patterns, but gateway sees all requests. Result: excessive one-shot sessions in distillation queue.

<!-- lore:019e1325-182f-7b30-bec0-ac4d383f3cc9 -->
* **One-shot stream=false sessions create session bloat**: OpenCode sub-agent calls (explore, code, etc.) via Task tool create stream=false one-shot sessions that appear as "new=true" in logs but never continue. Each sub-agent request gets treated as a separate new session with messages=1, creating session bloat. These sessions go through full temporal message storage and distillation discovery despite being ephemeral tool calls. Gateway lacks shouldSkip logic that opencode adapter has for child sessions. Pattern: every stream=false session appears exactly once with new=true, never seen again.

<!-- lore:019e1323-ffde-75c3-9af8-d52ac068fe87 -->
* **OpenCode distillation threshold skips layer 1 entirely**: OpenCode plugin only triggers backgroundDistill() for result.layer >= 2 (index.ts:1253), completely skipping layer 1 distillations. Combined with layer 1 not setting urgentDistillation flag, means layer 1 contexts never get background knowledge extraction. Layer 1 handles basic message thresholds but distillation never runs. Fix: change threshold to layer >= 1 and ensure layer 1 sets urgentDistillation flag.

<!-- lore:019e1325-a0eb-7fd6-ab3f-abfa591ed8d3 -->
* **Session fingerprinting breaks after gradient distillation**: Gateway identifies sessions by fingerprinting the first user message via \`identifySession()\`, but gradient transform at layer 1+ prepends distilled prefix and windows recent messages. The original first user message gets replaced in the transformed context sent to upstream. However, clients (opencode/Claude Code) maintain their own conversation history with original messages, so they re-send the same first user message on subsequent requests. Session identification should work as long as clients don't adopt the transformed context from model responses.

<!-- lore:019e1325-2a56-7cd5-8572-5eb9d43c8724 -->
* **Session fingerprinting breaks after gradient distillation transforms first user message**: Session identification uses \`fingerprintMessages()\` on first user message content, but gradient layer 1+ transforms prepend distilled prefix and window recent messages via \`tryFit()\`. This changes the first message seen by upstream LLM. However, fingerprinting happens on raw \`req.messages\` before gradient transform, so gateway-only usage preserves session continuity. Critical gotcha: opencode adapter's message transform hook modifies what opencode stores locally, potentially breaking session continuity on subsequent requests when opencode re-sends its stored conversation history.

<!-- lore:019e1325-a0f0-70d8-996a-97290575cbbb -->
* **Session fingerprinting relies on first user message unchanged by client**: Gateway session fingerprinting uses \`fingerprintMessages()\` on raw \`req.messages\` from client, hashing first user message text. Gradient transforms modify messages sent to upstream LLM but don't affect client's stored conversation history. Client re-sends same original first message each turn. However, session markers (\`\[lore:\<sessionID>]\`) are exported but never used - identification is purely fingerprint-based. Risk: if client echoes back transformed content or opencode adapter modifies stored messages, fingerprint breaks.

<!-- lore:019e1325-2a5a-733b-b62f-f6bf502f96f4 -->
* **Session fingerprinting survives gradient transforms in gateway mode**: Gateway computes session fingerprint from raw \`req.messages\` before gradient transform, but opencode plugin mode modifies stored messages. In gateway: client sends original messages → \`identifySession\` hashes first user message → gradient transform only affects upstream request → client retains original history → fingerprint persists across turns. In plugin mode: message transform hook (index.ts:1246) splices transformed messages into opencode's conversation array, potentially replacing first user message with distilled prefix. This breaks session continuity if opencode stores the modified messages as conversation history.

<!-- lore:019e1325-a0e6-7d80-ad36-5f244cba43cc -->
* **Session fingerprinting survives gradient transforms via timing**: Gateway session fingerprinting happens on incoming raw messages before gradient transform, not after. Client (opencode/Claude Code) always sends full conversation history with original first user message intact. Gateway transforms messages for upstream LLM but doesn't modify client's stored history. Fingerprint remains stable across requests even when gradient layer 1+ replaces first message with distilled prefix for LLM context. The timing separation (identify → transform → send upstream) preserves session continuity.

<!-- lore:019e1325-2a4b-790a-8b4a-e088a6da6bb9 -->
* **Session fingerprinting vulnerable to gradient layer transformations**: Session identification relies on \`fingerprintMessages()\` hashing the first user message content from \`req.messages\`. Gateway mode preserves session continuity because clients (Claude Code/opencode) maintain their own conversation history and re-send original messages unchanged. But opencode adapter mode modifies messages in-place via transform hooks, potentially changing the first user message when gradient layers apply distilled prefixes. This breaks session continuity as subsequent requests generate different fingerprints. Session markers \`\[lore:\<sessionID>]\` exist in codebase but are never actually used for identification.

### Pattern

<!-- lore:019e131f-a1bf-758c-85f8-7ff2811645ca -->
* **Temporal adapter abstracts session lifecycle across multiple backends**: Temporal adapter provides unified session lifecycle interface (\`recordMessage()\`, \`findUndistilledSessions()\`, \`markSessionDistilled()\`) across file-based (temporal-fs.ts) and SQLite (temporal-db.ts) backends. Both track \`distilled\` boolean but database version uses proper transactions for atomic operations.
