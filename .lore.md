### Gotcha

<!-- lore:019e1807-e55f-7f83-a56f-c5b299d226c0 -->
* **Bun segfault during NAPI cleanup when embedding worker left alive at test exit**: When running the full test suite, fire-and-forget \`embedKnowledgeEntry()\` calls in non-embedding test files (e.g. \`ltm.test.ts\`) spawn a shared worker. If \`resetProvider()\` is called synchronously in any test (especially the embedding test's \`isAvailable\` suite), it kills the worker mid-flight during NAPI cleanup, causing Bun to segfault at process exit. Fixes: (1) always \`await resetProvider()\` in test files, and (2) add a \`process.on('beforeExit')\` handler in the embedding module itself to shut down the worker cleanly before process exit.

### Pattern

<!-- lore:019e1808-59f7-7aae-99b1-dcee3320532f -->
* **.lore.md is the durable backup; DB entries are re-imported from it on session start**: Knowledge entries live in both the SQLite DB and per-project \`.lore.md\` files. The DB is the live store; \`.lore.md\` is the durable backup. On session start, the system re-imports \`.lore.md\` into the DB if the cache is clear. This means \`.lore.md\` in git is the true source of truth for recovery — restoring it from a git commit and clearing the lore file cache is sufficient to recover lost DB entries.
