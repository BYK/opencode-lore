### Gotcha

<!-- lore:019e182a-b4aa-7b3a-8d52-fdfcd6638552 -->
* **Binary smoke test: --check-embeddings path requires fastembed to load correctly**: Binary smoke test --check-embeddings requires fastembed worker in $bunfs. Three root causes and fixes: 1. \*\*Worker missing\*\*: Bun 1.3.13 silently drops extra entrypoints on macOS/Windows. Fix: \`new Worker(import.meta.url)\` from compiled binary re-enters same binary. Wrapper entrypoint checks \`isMainThread\`: true→gateway logic, false→embedding worker. 2. \*\*macOS NAPI crash\*\*: Bun 1.3.x panics on ONNX teardown at process exit. Fix: call \`\_exit(0)\` via \`bun:ffi\` to skip atexit handlers. 3. \*\*Windows path encoding\*\*: \`import.meta.url\` inside binary resolves to \`file:///B:/%7EBUN/root/wrapper.js\` (URL-encoded \`~BUN\`). \`new Worker()\` can't resolve this. Fix: \`decodeURIComponent()\` on the pathname before passing to \`new Worker()\`. macOS uses \`file:///$bunfs/root/wrapper.js\`. Both platforms resolve \`import.meta.url\` with \`.js\` extension.

<!-- lore:019e1759-b992-76f7-824e-318d5d206235 -->
* **extractModelFromMetadata must skip 'unknown' to find real model IDs**: When iterating conversation messages to extract model ID from metadata, user messages carry \`"modelID":"unknown"\`. Iterating chronologically hits user messages first and returns \`"unknown"\` instead of the real model (e.g. \`claude-opus-4-6\`). Fix: skip entries where the extracted value is \`"unknown"\` so iteration falls through to assistant messages with real model IDs.

<!-- lore:019e182a-b4a7-781f-81ef-91319a3d7d94 -->
* **Fastembed worker lifecycle: null vs undefined and race conditions in tryLoadFastembed**: (gotcha) Fastembed worker lifecycle: subtle bugs in tryLoadFastembed and worker ref management: (1) After shutdown, set \`cachedProvider = null\` (not \`undefined\`) so \`getProvider()\` doesn't re-spawn orphan workers. (2) Test suites must save/restore the live worker (\`\_saveAndClearProvider\`/\`\_restoreProvider\`) instead of killing it — avoids double ONNX loads causing segfaults. (3) \`tryLoadFastembed()\` must not over-aggressively re-check \`fastembedProbed\` after async boundary when upstream \`isAvailable()\` already set \`fastembedProbed=true, fastembedAvailable=false\`. (4) Worker created with \`unref()\` — implement \`updateWorkerRef()\` calling \`worker.ref()\` when \`pendingRequests.size > 0\`, \`worker.unref()\` when empty; call from \`embed()\` and all completion paths.

<!-- lore:019e1403-b62a-7944-8d5e-f6d70ddc2801 -->
* **Gradient layer resets indicate undetected Claude Code compaction events**: (gotcha) Gradient layer resets indicate undetected Claude Code compaction; false-positive interception stalls sessions: Gateway detects compaction via gradient layer drops (4→0) with system prompt changes or message count drops (~163K token threshold). Lore intercepts and generates a summary using COMPACT\_SUMMARY\_TEMPLATE (Goal/Constraints/Progress structure). False-positive bug: if \`isCompactionRequest()\` misclassifies a normal turn (e.g., gradient prefix contains text matching COMPACTION\_SYSTEM\_PATTERNS like 'anchored context summarization assistant'), gateway returns a compaction response. OpenCode displays the Goal/Progress summary to the user as a regular message, stalling the session. Model outputting the template verbatim is a symptom of false-positive interception, not hallucination. Fix: ensure no knowledge entries or distillations contain compaction-pattern phrases. Remove 'I'm ready to continue.' from COMPACT\_SUMMARY\_TEMPLATE.

<!-- lore:019e1859-3491-707b-b6bd-4ee73e9c1f1c -->
* **LoreAI project canonical path may point to \_\_tmp test dir after test runs**: After test runs, the opencode-lore project's canonical path in the DB can get set to the \`\_\_tmp\_agents\_file\_\_\` test directory instead of the real repo path (e.g. \`/home/byk/Code/opencode-lore\`). This causes \`.lore.md\` auto-import to silently skip the real project. Fix: check \`SELECT path FROM projects\` and update the canonical path + remove stale aliases via \`DELETE FROM project\_aliases WHERE path LIKE '%\_\_tmp%'\`.

* **OpenCode slash commands require SSE streaming when stream:true set**: OpenCode/Claude Code requests with stream:true expect SSE responses, not JSON. Plain JSON responses get dropped silently. Use streamHttpResponse() with buildSSETextResponse() for streaming clients, jsonResponse() for non-streaming. Check req.stream boolean to determine format. Essential for slash command responses and synthetic intercepts.

* **Recall tool marker shows empty query when id parameter is used**: (gotcha) Recall tool marker shows empty query when id parameter is used: The \`recall\` tool accepts an \`id\` parameter for direct lookup, leaving \`query\` as empty string \`""\`. The SSE marker builder uses \`input.query\` unconditionally, producing \`📚 Searching all archives for ""…\`. Fix: when \`id\` is present, build a different marker (e.g. \`📚 Looking up entry \<id>…\`) in both the pipeline's tool execution path (~line 768 of pipeline.ts) and \`replaceRecallWithMarker\` (~line 425).

<!-- lore:019e186e-44ec-74e2-a4cd-0d33ee8d87cb -->
* **Search AND→OR cliff: progressive relaxation pattern for FTS queries**: (gotcha) Search AND→OR cliff: progressive relaxation pattern for FTS queries: All FTS search functions (temporal.ts, ltm.ts, lat-reader.ts) had a hard AND→OR fallback — if AND returns nothing, retry with any single term, producing 89+ results for multi-word queries. Fix: \`ftsQueryRelaxed()\` in \`search.ts\` generates a cascade dropping one term at a time before falling to OR. \`runRelaxedSearch()\` utility wraps a query runner and tries each variant until results found. All callers must be updated to use \`runRelaxedSearch\` instead of the manual two-step AND/OR pattern. Additionally, \`searchRecall()\` RRF has no output cap — must cap at ~30 results. The UI's \`formatSearchResult()\` must also truncate content (use \`mdTruncated()\` helper) and apply a relevance score floor to avoid rendering entire distillation sessions.

<!-- lore:019e16c9-f53f-739f-b9f1-c825aec1d375 -->
* **Subagent turns must skip temporal storage to prevent distillation contamination**: (gotcha) Subagent turns must skip temporal storage and background work to prevent distillation contamination: Subagent turns must skip temporal storage (pipeline.ts ~lines 1196-1217), session state updates (~1277-1290), and background work scheduling (~1294). Subagent output becomes a tool result in the parent conversation, inflating parent inputTokens and potentially triggering compaction. Compaction should be blocked during plan mode. Post-subagent false-positive compaction: after subagents complete, \`isCompactionRequest\`/\`isStructuralCompaction\` can misfire on the tool-result turn, causing gateway to return a GatewayResponse with \`inputTokens:0\` and the model outputs Goal/Progress/Key Decisions structure as a normal assistant message (\`summary\` field NOT set). Debug: check if gateway returns compaction response for a non-compaction turn.

* **Test sessions with \_\_tmp paths contaminate distillation cost analytics**: Distillation cost analysis is skewed by sessions from \`\_\_tmp\_agents\_file\_\_\` test projects. Always filter projects whose path contains \`\_\_tmp\` when computing production cost metrics. ### Pattern

<!-- lore:019e1759-b9a3-7705-b678-bc8be020e18c -->
* **Avoided compaction cost must include cache-bust penalty on conversation model**: Compaction savings calculations must include the cache-bust penalty: after each compaction, ~30K tokens of post-compaction context must be re-written to prompt cache at \`cache\_write\` rates on the conversation model (e.g. Opus at $18.75/M vs $1.50/M cache\_read). Omitting this undervalues each avoided compaction ~3.6x ($0.19 → $0.71). \`estimateCompactionCost\` requires both \`workerModel\` and \`conversationModel\` parameters; \`updateShadowContext\` and pipeline callsites must pass \`req.model\` as the conversation model.

<!-- lore:019e16d2-2d7f-70a5-8f17-038b85cc11c5 -->
* **Cache warmer logging pattern for prompt cache telemetry**: Cache warming requests log comprehensive telemetry: input\_tokens (total request size), cache\_read\_tokens (cached prefix), cache\_write\_tokens (new content), hit percentage, and estimated cost. Format: 'cache-warmer: ✓ refresh session=... input=85000 cacheRead=84500 hit=99% cost=$0.0127'. Status symbols: ✓ (full hit), ~ (partial), ✗ (uncached). Essential for monitoring cache efficiency and costs in warmup operations.

Entries with known UUIDs update in place; entries without UUIDs get new UUIDv7s.
