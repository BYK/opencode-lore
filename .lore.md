## Long-term Knowledge

### Architecture

<!-- lore:019e0238-28bb-72e4-8551-f308662d86c0 -->
* **Embedding backfill unified at startup across all hosts**: Unified embedding backfill system with config migration: Moved embedding backfill from OpenCode-only to unified \`runStartupBackfill()\` in core. Both OpenCode and Pi call this at init, ensuring equal knowledge + distillation embedding coverage. Embedding config fingerprint enables zero-downtime provider migration via \`checkConfigChange()\` comparing current to stored fingerprint (\`provider:model:dimensions\`). On mismatch, stale embeddings auto-clear and full backfill regenerates. Enables seamless Voyage (1024d) ↔ Local (384d) ↔ OpenAI (1536d) migrations. Coverage stats logged to stderr.

<!-- lore:019e03da-045c-7ce9-8fae-0f01de363b70 -->
* **Gateway as universal HTTP integration point**: Gateway accepts Anthropic \`/v1/messages\` protocol, applies full Lore pipeline (gradient + LTM + distillation + curation), forwards to upstream. Works for any HTTP client by setting base URL. Enables recall tool interception universally without per-host plugins. Session tracking via base62 markers injected into assistant responses. Compaction requests intercepted to trigger distillation.

<!-- lore:019e0381-8263-71ec-b845-06b715363ce2 -->
* **Gateway recall interception — two-case stream-and-intercept design**: Gateway recall interception with two-case stream design: Gateway transparently intercepts recall tool\_use blocks mid-response using \`RecallAwareStreamAccumulator\`. Case 1 (recall-only): suppress recall from client stream, execute via follow-up, inject '\[Searching memory...]' text, continue with re-indexed blocks in same response. Case 2 (recall + other tools): suppress recall, execute in parallel, store result in \`sessionState.pendingRecall\` with 60s TTL, inject \`{ toolUseId, position, result, timestamp }\` into next request's assistant message via \`injectPendingRecall()\`. Logs show execution paths for debugging.

<!-- lore:019e041d-d9b6-7cc6-88ff-82ed24478d48 -->
* **Layer-1 message window construction with budget-based trimming**: Layer-1 gradient uses \`tryFitStable()\` but busts ~98% due to budget instability. Root cause: pinned windows cached at budget X fail validation when current budget becomes Y. Budget fluctuates from: 1) Global \`calibratedOverhead\`/\`ltmTokens\` cross-session contamination, 2) \`deduplicateToolOutputs()\` changing token estimates mid-window, 3) Fingerprint changes from raw window cutoff shifts. Fix requires budget snapshot in cache plus hysteresis for noise absorption.

<!-- lore:019e0425-2e67-77d5-8091-4d228c43db80 -->
* **Layer-1 raw message window construction via backward token scan**: Layer-1 raw window built by \`tryFitStable()\` (lines 1159-1232) using backward token scan. Splits budget: 25% distilled, 40% raw. Current turn always fully included (\`currentTurnStart()\` to end), then backward scan from most recent older messages until \`remainingBudget = rawBudget - currentTurnTokens\` exhausted. Cutoff drops oldest messages first. Three mechanisms cause leading-edge ID shifts: 1) \`currentTurnStart()\` boundary moves as tool-chains extend, 2) Pinned window cache overflow triggers fresh scan when appended messages exceed budget, 3) \`deduplicateToolOutputs()\` changes token estimates of older messages between transforms.

<!-- lore:019e0427-0279-72c7-b0b3-3fcd1da83c04 -->
* **Three-part solution for gradient cache-bust stabilization**: Three-part solution for 97-98% cache-bust stabilization: 1) Snapshot \`rawBudget\` in \`RawWindowCache.pinnedBudget\` at pin creation to prevent pin checks from using different budget than pin creation, 2) Hysteresis (10%) in \`tryFitStable\` pin validity (\`pinnedTokens <= pinnedBudget \* 1.10\`) to absorb dedup token estimate variance, 3) Move \`ltmTokens\` to per-session \`SessionState\` with optional \`sessionID\` parameter in setter/getter. Combination eliminates cross-session contamination, budget drift, and estimation noise simultaneously.

### Pattern

<!-- lore:019e0428-694b-7aab-a42a-1ce98c4c3b1f -->
* **RawWindowCache requires pinnedBudget snapshot for stability**: \`RawWindowCache\` stores \`pinnedTokens\` and \`pinnedBudget\` (budget snapshot at pin creation). Pin validity check uses 10% hysteresis: \`pinnedTokens <= pinnedBudget \* 1.10\` instead of exact match against current budget. Decouples pin validity from volatile budget fluctuations caused by dedup token estimate changes and overhead drift. Absorbs ~8-17K token noise on ~47K budget without false evictions.

<!-- lore:019e0428-6964-7feb-b42b-d94e1cfb6aa6 -->
* **SessionState per-session isolation pattern for globals**: Move problematic module-level globals to \`SessionState\` for per-session isolation. Pattern: add optional \`sessionID\` parameter to setter functions. Store on session state if ID provided, else use module-level fallback for backward compatibility. Read from session state first in transforms. Example: \`setLtmTokens(tokens, sessionID?)\` stores in \`sessState.ltmTokens\` if sessionID provided. \`transformInner\` reads \`sessState.ltmTokens\` instead of global, eliminating cross-session contamination where one session's LTM pollutes another's budget calculations.

<!-- lore:019e0427-0281-75c6-bd89-e27b4e64a72f -->
* **Three-part fix for tryFitStable cache instability**: Fix tryFitStable cache busts with complementary changes: 1) Add \`pinnedBudget: number\` to \`RawWindowCache\` to snapshot budget at pin creation. 2) Use 5% hysteresis buffer in pin validation: \`totalTokens <= pinnedBudget \* 1.05\` to absorb token estimation noise. 3) Move \`ltmTokens\` from module global to \`SessionState.ltmTokens\` and update \`setLtmTokens(sessionID, tokens)\` signature. All three address different instability sources: budget snapshots prevent pin-check budget mismatches, hysteresis absorbs dedup estimation variance, per-session state eliminates cross-session contamination.

<!-- lore:019e0428-69dc-733b-8292-0775d1a1f6a4 -->
* **Three-part fix pattern for tryFitStable cache stability**: Comprehensive fix for tryFitStable cache busts uses three complementary approaches: 1) Budget snapshot - store \`pinnedBudget\` in \`RawWindowCache\` during cache write, use it (not current budget) in pin validity check, 2) Hysteresis buffer - use \`rawBudget \* 0.95\` threshold in pin check to absorb estimation noise from dedup changes, 3) Per-session isolation - move \`ltmTokens\` to \`SessionState\` to eliminate cross-session contamination. All three address different instability sources: budget drift, estimation variance, and shared globals. Pattern prevents 97%+ cache bust rates in tool-heavy sessions.

* **Three-pronged fix for tryFitStable cache instability**: Three-pronged fix for tryFitStable cache instability: 1) Snapshot budget in \`RawWindowCache.pinnedBudget\` at pin creation time, 2) Hysteresis with 5% tolerance in pin overflow check (\`pinnedTokens > pinnedBudget \* 1.05\`) to absorb dedup estimation noise, 3) Move module-level \`calibratedOverhead\` and \`ltmTokens\` to per-session state to prevent cross-session contamination. The \`setLtmTokens(sessionID, tokens)\` API requires sessionID parameter - available as \`input.sessionID\` in adapters. All three address different aspects: budget drift, estimation variance, and session isolation.
